# Rebuild Challenges

This project involved more than creating a new website layout. It required recovering from a legacy website failure, deciding what could be preserved, and rebuilding the site into a maintainable structure.

## 1. Starting From Limited Administrative Access

The original website had already been created by someone else. My initial role was limited to routine website administration, such as uploading photos, sermon videos, and basic content updates through an administrator account.

This meant the rebuild did not begin with full design ownership from day one. It began as a maintenance responsibility that later became a full rebuild project after the legacy system became unreliable.

## 2. Legacy CMS Limitations

The original site was based on XE, a legacy CMS that became increasingly difficult to maintain over time.

The main issue was not only the visual layout, but the long-term maintainability of the system. As the CMS ecosystem became outdated, it became harder to rely on the old structure for a public-facing organization website.

## 3. Hosting and Platform Changes

After hosting or platform environment changes, the old website structure became unreliable. The exact technical cause was difficult to isolate from the outside, but the result was clear: the existing site could no longer be maintained confidently in its previous form.

Rather than continuing to patch an unstable legacy setup, the decision was made to rebuild the website using a more maintainable platform.

## 4. Rebuild vs. Repair Decision

One of the main challenges was deciding whether to repair the old website or rebuild it from the ground up.

The rebuild approach was chosen because it allowed the site structure, content organization, media workflow, and future maintenance process to be redesigned instead of simply restoring the old layout.

## 5. Reorganizing the Site Structure

The rebuild required reorganizing the website around real visitor and content-management needs.

Key areas included:

- Public homepage structure
- Organization introduction pages
- Sermon and media access
- Photo gallery organization
- Community-related content
- Navigation menu hierarchy
- Live translation access path

The goal was to make the website easier for visitors to understand and easier for future maintainers to update.

## 6. Media and Gallery Management

The website needed to support recurring media updates, including photos, sermon videos, and event-related content.

This required creating a structure that could handle ongoing updates without requiring a redesign each time new media was added.

## 7. Translation Access Integration

The rebuilt website later needed to provide a clear access point for live translation resources.

The translation system was kept separate from the main website so that the public website could remain stable while the translation workflow could continue to evolve independently.

## 8. Privacy and Production Safety

Because this was a real public-facing organization website, the portfolio version of this project had to be sanitized.

This repository intentionally avoids including:

- Administrator credentials
- Hosting account details
- Server configuration
- Internal records
- Private media files
- Production WordPress settings

The focus is on documenting the rebuild process, structure, and decision-making rather than exposing production materials.

## Summary

The main challenge of this project was not simply building pages. It was turning an unstable legacy website situation into a more maintainable WordPress-based structure that could support public information, recurring media updates, and future service-related integrations.
